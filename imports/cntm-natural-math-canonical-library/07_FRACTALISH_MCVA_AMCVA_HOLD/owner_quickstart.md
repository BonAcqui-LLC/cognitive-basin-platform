# Owner Quickstart

If this project feels bigger than your available energy right now, use this page.

You do not need to master GitHub, video, branding, community management, and software release all at once.

For `v0.1-alpha`, the goal is much smaller:

- make the work publicly readable,
- make the core docs downloadable,
- make the code visible,
- let people suggest improvements,
- and keep control of what gets merged.

## The Simple Mental Model

There are only three public surfaces you need to think about:

### 1. Fractalish site

This is the human-facing doorway.

Use it to answer:

- what is this,
- why does it matter,
- where should I start,
- what does it not claim.

### 2. `mcva` public repo

This is the working public lab notebook.

Use it to hold:

- the whitepaper,
- the glossary,
- the roadmap,
- the open doctrine,
- simple code,
- tests,
- issue tracking,
- and contribution flow.

### 3. Private/internal material

This stays out of public release unless it is cleaned and intentionally approved.

Keep private:

- `mcva-gate`,
- private prompts,
- legal drafts,
- medical details,
- financial or personal records,
- secrets and keys,
- unclear-license datasets,
- and unfinished internal strategy notes.

## What Contributors Actually Do

Public contributors usually do one of four things:

- open an issue,
- fork the repo,
- make changes in their fork,
- open a pull request asking you to merge them.

They cannot silently overwrite your main repo.

Your safe default rule:

**Do not merge anything until we review it together.**

That is enough. You do not need to be fast.

## The Lowest-Stress GitHub Workflow

Use this workflow until you feel stronger:

1. Make the repo public.
2. Allow issues.
3. Allow pull requests.
4. Do not give broad write access.
5. Treat every external change as a review item, not as an obligation.
6. Ask for help reviewing before merging anything uncertain.

If someone opens:

- an **issue**, they are starting a discussion.
- a **pull request**, they are proposing a change.
- a **fork**, they are making their own copy.

Only a merged pull request changes your main public repo.

## What Can Wait

These do **not** have to be solved before `v0.1-alpha`:

- a polished video,
- a finished atlas,
- a perfect data pipeline,
- a final descriptor language,
- a complete benchmark suite,
- a hosted production platform,
- a broad contributor community,
- or a perfect visual brand system.

## What To Say Publicly

Use this line consistently:

> The core framework is open. If this work helps you, support it, cite it, contribute back, sponsor development, or hire us to apply it.

And keep these boundaries visible:

- this is a research framework,
- not a diagnosis engine,
- not a universal law,
- not a regulatory certifier,
- not a replacement for domain experts.

## Good First Public Actions

If you only have enough energy for a few actions, do these:

1. Publish the public repo.
2. Put the `Start Here` path on the site.
3. Post one short announcement.
4. Invite issues, not expectations.
5. Review contributions slowly.

## When You Feel Overwhelmed

Return to this rule:

You are not trying to launch a finished empire.
You are trying to publish a credible public research starting point.

That is a much smaller job.
